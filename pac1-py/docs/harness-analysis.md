# HARNESS-анализ: агент pac1-py

> Agent = Model + Harness. Harness -- инфраструктура, управляющая работой агента.
> Дата: 2026-04-08 | Ветка: dev | Скор: ~90%

## Методология

HARNESS-подход оценивает инфраструктуру агента по двум осям:
- **Направляющие** (feedforward) -- управление *до* действия
- **Датчики** (feedback) -- наблюдение *после* действия, самокоррекция

Плюс сквозные аспекты: песочница, тестирование, модульность, наблюдаемость, воспроизводимость.

## Сильные стороны

### 1. Направляющие (Feedforward) -- Оценка: A

| Механизм | FIX | Описание |
|----------|-----|----------|
| Нормализация инъекций | FIX-203 | zero-width символы + NFKC + leet-speak маппинг перед маршрутизацией |
| Семантический роутер | FIX-204 | LLM классифицирует маршрут (EXECUTE/DENY/CLARIFY/UNSUPPORTED) до входа в loop |
| Защита области записи | FIX-250 | Код блокирует запись в AGENTS.MD, docs/channels/ (кроме otp.txt) |
| Format Gate | FIX-214 | Inbox без заголовка From:/Channel: -> OUTCOME_NONE_CLARIFICATION |
| Порядок проверки инъекций | FIX-235 | Проверка инъекций всегда выполняется ПЕРВОЙ, затем format gate |
| Модульные промты | -- | Тип задачи -> конкретный набор блоков (_CORE, _EMAIL, _INBOX, _DELETE) |
| Динамический Prompt Builder | -- | LLM-аддендум с правилами по задаче (подсчёт, даты, контакты) |

**Почему сильно:** 7 механизмов feedforward обеспечивают эшелонированную защиту. Задачи фильтруются, классифицируются и ограничиваются *до* первого действия агента. Regex fast-path обрабатывает высоковероятные типы без LLM-вызовов.

### 2. Датчики (Feedback) -- Оценка: B+

| Механизм | FIX | Описание |
|----------|-----|----------|
| Детекция зацикливания | FIX-74 | 3 сигнала: fingerprint 3x (deque maxlen=6), повторная ошибка, 6 шагов без мутации |
| Evaluator Gate | FIX-218 | Отдельная LLM проверяет outcome vs evidence до report_completion |
| Контаминация email | FIX-206 | Post-write проверка: пути vault утекли в outbox -> блокировка |
| Security Interceptor | FIX-253 | Принудительное применение DENIED_SECURITY на отмеченных задачах |
| Format Gate Enforce | FIX-259 | Принудительное CLARIFICATION + bypass evaluator при срабатывании format gate |
| Авто-заполнение grounding | FIX-232 | Автосбор contacts/accounts из step_facts для lookup/inbox |
| Компактификация лога | -- | prefix + последние 5 пар + дайджест (управление ростом контекста) |

**Почему сильно:** Паттерн Evaluator соответствует подходу Anthropic 3-agent harness (отдельный агент работает, отдельный агент оценивает). Детекция зацикливания не зависит от типа задачи и покрывает 3 независимых сигнала.

### 3. Песочница безопасности -- Оценка: B+

| Слой | Детали |
|------|--------|
| Песочница кода | Только datetime/json/re/math; заблокированные builtins (eval, exec, open, __import__); SIGALRM 5s |
| Защищённые пути | AGENTS.MD, docs/channels/ неизменяемы (кроме otp.txt для OTP-потока) |
| Защита удаления | Всегда list -> по одному -> повторный list; никогда wildcard; никогда `_`-префиксные |
| 8 Security Gates | FIX-203, 206, 214, 215, 218, 235, 253, 259 |
| Уровни доверия Inbox | blacklist/admin/valid/non-marked с исключением для admin |

### 4. Мульти-модельная маршрутизация -- Оценка: A

- 8 типов задач (default, think, longContext, email, lookup, inbox, distill, coder) -> специализированные модели
- 3-уровневый LLM fallback: Anthropic SDK -> OpenRouter -> Ollama
- Детекция capabilities с файловым кешем (~/.cache/capability_cache.json, TTL 7 дней)
- Regex fast-path для высоковероятных типов пропускает LLM-классификацию
- Кеш маршрутов: sha256(task_text[:800]) -> детерминированный в рамках запуска

### 5. Наблюдаемость -- Оценка: B

- Per-call трекинг токенов (input/output/thinking для Anthropic, eval_count/eval_ms для Ollama)
- Метрики evaluator (call_count, total_ms)
- Дублирование логов в файлы с удалением ANSI (logs/{ts}_{model}/)
- Сводная таблица с throughput (tok/s) по моделям

## Слабые стороны

### 1. Тестирование -- Оценка: D

Тесты есть (4 файла в tests/), но покрытие минимально:
- `test_classifier.py` -- юнит-тесты классификатора
- `test_json_extraction.py` -- тесты извлечения JSON
- `test_capability_cache.py` -- тесты кеша capabilities
- `conftest.py` -- фикстуры

**Отсутствует:**
- Тесты security gates (FIX-203, 206, 214, 215, 235, 253, 259)
- Тесты потока evaluator
- Тесты детекции зацикливания
- Тесты сборки промтов
- Интеграционные тесты с мокированным PCM runtime
- Регрессионные тесты для отдельных FIX-ов

**Риск:** Security gates -- самый критичный код с нулевым покрытием тестами. Любой рефакторинг может молча сломать их.

### 2. Мокирование инструментов -- Оценка: F

Нет mock-инфраструктуры для:
- LLM-вызовов (call_llm_raw) -- нет способа тестировать с детерминированными ответами
- PCM dispatch -- нет способа симулировать операции vault
- Сетевых ошибок, rate limits, некорректных ответов

Единственный путь валидации -- запуск полного бенчмарка (`make run`).

### 3. Управление циклом -- Оценка: B-

Детекция зацикливания работает, но распределение бюджета жёсткое:
- **Нет константы максимума шагов** -- цикл ограничен только TASK_TIMEOUT_S (по умолчанию 180s)
- Нет адаптивного таймаута на основе сложности задачи
- Нет раннего выхода при высокой уверенности в решении
- Нет распределения бюджета токенов по типам задач
- Stall hint -- только текстовая инъекция, нет принудительной смены стратегии

### 4. Модульность -- Оценка: C+

**Хорошо:** Промты модульные; classifier/evaluator/builder -- отдельные компоненты.

**Плохо:**
- `loop.py` -- **2604 строки**. Содержит: маршрутизацию, 8 security gates, извлечение JSON (7-уровневый fallback), компактификацию лога, детекцию зацикливания, интеграцию evaluator, grounding, dual-write enforcement, cross-account check, format gate. Каждый FIX добавляет ещё один interceptor сюда.
- `dispatch.py` -- **739 строк**. Содержит: 3 LLM-бэкенда, песочницу кода, coder sub-agent, детекцию capabilities, загрузку секретов, защищённые пути.
- Высокая связность: loop.py напрямую знает про внутренности evaluator, формат промтов, ошибки dispatch.

### 5. Смещение обработки ошибок -- Оценка: C

Всё работает в режиме fail-open:

| Компонент | При ошибке |
|-----------|-----------|
| Evaluator | -> approved=True |
| Prompt Builder | -> "" (пустой аддендум) |
| Router | -> CLARIFY fallback |
| Capability probe | -> считать поддерживаемым |
| Извлечение JSON | -> 7-уровневый fallback |

**Хорошо для доступности**, но:
- Сломанный evaluator = тихая деградация качества (quality gate отсутствует полностью)
- Нет метрик по частоте fallback -> тихая деградация невидима
- Нет алертов при превышении порога частоты fallback

### 6. Воспроизводимость -- Оценка: D

- Нет механизма записи/воспроизведения сессий
- Нет контроля seed для Anthropic API (ограничение платформы)
- Кеш маршрутов только в памяти (в рамках одного запуска)
- Невозможно воспроизвести конкретную упавшую задачу офлайн для отладки
- Лог-файлы захватывают вывод, но не полные трейсы запрос/ответ

### 7. Адаптивный бюджет -- Оценка: D

- Статический TASK_TIMEOUT_S=180s для всех задач
- Нет динамического распределения шагов (простой lookup получает тот же бюджет, что и сложный inbox)
- Только thinking_budget зависит от модели (через конфиг), но не от задачи
- Нет разделения бюджета токенов между фазами exploration и mutation

## Топ-3 рекомендации

### Приоритет 1: Тесты security gates

Добавить юнит-тесты для каждого security interceptor. Это самые критичные пути кода с нулевым покрытием:
- Нормализация инъекций (FIX-203): тесты leet-speak, zero-width, unicode
- Format gate (FIX-214): тесты детекции From:/Channel: и граничных случаев
- Защита области записи (FIX-250): тесты enforcement защищённых путей
- Inbox-инъекции (FIX-215): тесты классификации уровней доверия

### Приоритет 2: Декомпозиция loop.py

Вынести в отдельные модули:
- `agent/security.py` -- все security interceptors (FIX-203/214/215/235/253/259)
- `agent/json_extract.py` -- 7-уровневая логика извлечения JSON
- `agent/compaction.py` -- компактификация лога + построение дайджеста
- `agent/stall.py` -- сигналы детекции зацикливания

Цель: loop.py менее 800 строк, сфокусированный только на оркестрации.

### Приоритет 3: Механизм replay

Логировать полный трейс по задаче: `{task_text, route, steps: [{llm_request, llm_response, dispatch_result}], final_outcome}` в JSON. Это даёт:
- Офлайн-отладку упавших задач
- Регрессионное тестирование без живых LLM-вызовов
- Профилирование производительности по шагам
